## Описание проекта

`NotesApp` — Web API приложение для работы с заметками. В проекте есть категории и заметки, связанные отношением один-ко-многим. Данные хранятся в SQLite базе `notesapp.db`, доступ к данным вынесен в репозитории.

## Repository

Repository отделяет контроллеры от прямой работы с `DbContext`. Контроллер отвечает за HTTP-запросы и ответы, а репозиторий выполняет запросы к базе данных, фильтрацию, сортировку, создание, обновление и удаление записей.

В проекте используются `ICategoryRepository`, `CategoryRepository`, `INoteRepository` и `NoteRepository`. Благодаря интерфейсам контроллер зависит от контракта, а не от конкретной реализации.

## Что изучено

| Концепция | Описание |
| --- | --- |
| Repository | Изоляция логики доступа к данным |
| Interface | Контракт для репозитория |
| AddScoped | Регистрация зависимости на один HTTP-запрос |
| HasOne / WithMany | Настройка связи один-ко-многим |
| Include | Загрузка связанных данных |
| DeleteBehavior.Restrict | Запрет удаления категории с заметками |
| HasIndex | Индекс для ускорения запросов |
| Select DTO | Возврат клиенту только нужных полей |
| ApiResponse | Единый формат успешных ответов |
| ApiError | Единый формат ошибок |
| Data Annotations | Валидация входных данных |

## Главные выводы

1. Паттерн Repository помогает держать контроллеры проще и не смешивать HTTP-логику с запросами к базе.
2. Data Annotations одновременно валидируют входные данные и задают ограничения для EF Core.
3. Единый формат ответа удобен для фронтенда, потому что клиент всегда получает `success`, `message`, `data` или `errors`.
4. `DeleteBehavior.Restrict` защищает заметки от случайного удаления вместе с категорией.
5. `Include()` и проекция в DTO через `Select()` позволяют получить связанные данные без лишних запросов.
